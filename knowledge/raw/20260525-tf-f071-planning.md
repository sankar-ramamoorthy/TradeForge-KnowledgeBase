# Chosen Approach

Update `generate_thesis_review` to use the canonical event-store port:

```python
event_store.read_events()
```

Then filter the returned events to those that reference the requested decision.
The route already scans the latest thesis event and builds a `ThesisArtifact`;
that behavior remains intact.

# Rationale

The canonical event-store protocol supports append-only writes and
deterministic replay reads. Adding or relying on `load()` would create a second
read contract not represented in the domain port. Filtering after
`read_events()` matches existing runtime projection and API patterns.

# Affected Runtime Layer

- App layer: advisory API route context reconstruction.
- Tests: focused API regression coverage.

# Event, Replay, And Lifecycle Impact

- Event impact: none; no event schema or append behavior changes.
- Replay impact: positive; route uses deterministic replay history.
- Lifecycle impact: none; advisory review cannot transition lifecycle state.

# Testing Approach

Add a regression test that injects an event store implementing `read_events()`
and `append()` only, with no `load()` method. The test should call
`POST /advisory/thesis-review`, assert advisory output is returned, assert the
provider was invoked once, and assert no event append occurred.

Retain missing-thesis behavior as a controlled `404`.

# ADR Checkpoint

No ADR is required because this is a bug fix within existing event-store and AI
advisory boundaries.
