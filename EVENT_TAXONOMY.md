# TradeForge — Event Taxonomy

## Purpose

This document defines the canonical event model for TradeForge.

Events are the foundation of:
- system truth
- lifecycle reconstruction
- replayability
- auditability
- workflow integrity

All system state must be derivable from events.

---

# Core Principle

## Events Are Facts, Not Interpretations

Events represent something that **happened**.

They are NOT:
- opinions
- summaries
- forecasts
- AI-generated insights
- derived metrics
- UI states

If it did not happen, it is not an event.

---

# Event System Overview

TradeForge uses an **append-only event system**.

All meaningful system changes are recorded as immutable events in the Event Ledger.

Events enable:
- deterministic replay
- auditability
- workflow reconstruction
- behavioral analysis
- decision review

---

# Event Structure (Conceptual)

All events follow a consistent conceptual structure:

- event_type
- timestamp
- persona_id
- workspace_id (if applicable)
- entity_reference(s)
- payload (structured data)
- provenance metadata

Events are immutable once written.

---

# Top-Level Event Categories

TradeForge events are grouped into canonical domains:

```text
persona.*
workspace.*
market.*
scenario.*
decision.*
execution.*
review.*
system.*
```

Each category defines a bounded event space.

---

# 1. Persona Events

Events related to operator context and mode.

Examples:

- PersonaCreated
- PersonaActivated
- PersonaDeactivated
- PersonaSettingsUpdated

Purpose:
Define operational reasoning context.

Persona events affect interpretation of all downstream systems.

---

# 2. Workspace Events

Events related to persona-scoped operational environments.

Examples:

- WorkspaceCreated
- WorkspaceOpened
- WorkspaceClosed
- WorkspaceContextUpdated

Purpose:
Track the operational environment in which decisions occur.

Workspaces are cognitive contexts, not UI containers.

---

# 3. Market Events

Events representing observed market state changes.

Examples:

- MarketDataReceived
- PriceUpdated
- VolumeSpikeDetected
- VolatilityRegimeChanged
- MacroEventDetected

Important constraint:
Market events are observations, not interpretations.

---

# 4. Scenario Events

Events related to opportunity and risk generation.

Examples:

- ScenarioGenerated
- ScenarioRanked
- ScenarioInvalidated
- ScenarioPromotedToWatchlist

Important constraint:
Scenario events are **advisory constructs**, not decisions.

They represent system-generated hypotheses.

---

# 5. Decision Events

Events representing human decision workflow state.

Examples:

- TradeIdeaCreated
- ThesisCreated
- PlanCreated
- PlanApproved
- PlanRejected
- DecisionQueued
- DecisionExecuted

This is the **core workflow domain** of TradeForge.

Decision events are:
- explicit
- user-driven
- lifecycle-controlled

---

# 6. Execution Events

Events representing interaction with external systems.

Examples:

- OrderSubmitted
- OrderModified
- OrderCancelled
- FillReceived
- PositionOpened
- PositionClosed

Important constraint:
Execution events reflect external reality (broker/system state), not internal intent.

---

# 7. Review Events

Events related to reflection and learning.

Examples:

- ReviewStarted
- ReviewCompleted
- OutcomeEvaluated
- RuleViolationDetected
- BehavioralInsightRecorded

Purpose:
Support learning, adaptation, and replay analysis.

Review events are first-class system artifacts.

---

# 8. System Events

Events related to system operations.

Examples:

- SystemInitialized
- ProjectionRebuilt
- CacheInvalidated
- DataSyncCompleted

Purpose:
Infrastructure and operational observability.

---

# Event Lifecycle Model

Events follow strict lifecycle constraints:

## Creation
- Events are generated by system components or user actions
- Must include full context at time of creation

## Storage
- Events are appended to Event Ledger
- Never modified after creation

## Consumption
- Events are consumed by projections
- Used for UI rendering and analytics
- Used for replay reconstruction

---

# Event vs State Distinction

## Events (Canonical Truth)

- immutable facts
- historical record
- source of replay
- audit trail

Examples:
- OrderFilled
- PlanApproved
- ScenarioGenerated

---

## State (Derived)

- computed from events
- transient representation
- UI optimization layer

Examples:
- current positions view
- watchlist ranking
- dashboard summaries

State is NEVER authoritative.

---

# Invalid Event Patterns

The following are NOT valid TradeForge events:

- “MarketLooksBullish”
- “TraderConfident”
- “GoodSetupDetected”
- “HighQualityTrade”

Reason:
These are interpretations, not observable facts.

---

# Valid Event Criteria

A valid event must:

- represent something that happened
- be objectively observable (by system or user action)
- be timestamped
- be immutable
- be replayable
- be contextually grounded

---

# Event Design Principle

If an event requires interpretation to validate its existence, it is not a valid event.

Events must be:
> fact-first, interpretation-later

---

# Event Granularity Principle

Events should be:
- fine-grained enough to support replay
- not so granular that they become noise

Guideline:
- prefer atomic workflow transitions
- avoid aggregating multiple decisions into a single event

---

# Replay Dependency Rule

Replay systems must depend ONLY on:

- event ledger
- deterministic rules
- historical snapshots (optional optimization)

Replay must NOT depend on:
- live market APIs
- AI outputs
- UI state
- derived projections

---

# AI and Events

AI systems MAY:

- suggest potential events
- propose interpretations
- cluster event patterns

AI systems MAY NOT:

- directly write canonical decision or execution events
- override lifecycle events
- fabricate historical events

All AI suggestions must be reviewed through deterministic system layers before becoming events.

---

# Event Naming Convention

Events should follow:

- Past tense naming
- Explicit action description
- Domain prefix grouping

Examples:

- PlanApproved
- OrderSubmitted
- ScenarioGenerated
- ReviewCompleted

Avoid:
- vague naming
- emotional language
- interpretive descriptors

---

# Final Principle

Events are the atomic truth layer of TradeForge.

If an event is incorrect:
- downstream projections are incorrect
- replay is incorrect
- system memory is incorrect

Therefore:

> Event design is system design.
