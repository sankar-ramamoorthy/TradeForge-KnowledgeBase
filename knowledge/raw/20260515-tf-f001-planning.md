# TF-F001 Planning Capture

## Diagnosis Summary

Root cause is implementation incompleteness, not an architectural gap.

Thesis revision exists end-to-end. Plan revision was never built.
The thesis revision stage check is also too restrictive — it only allows revision from THESIS stage,
blocking the operator from revising the thesis once they've moved to the Plan Review workspace.

Two gaps:
1. Plan revision — entirely missing (event, endpoint, modal, button)
2. Thesis revision from Plan stage — blocked by strict stage check

## Approach

Mirror the thesis revision pattern for plan. Relax thesis stage check.

No ADR required — enhancement only, no lifecycle state changes, no event model structural changes.

## Scope Boundary

In scope:
- POST /lifecycle/decisions/revise-plan (new endpoint)
- GET /lifecycle/decisions/{id}/plan/history (new endpoint)
- Update GET /lifecycle/decisions/{id}/plan to return latest (created or revised)
- PlanRevisionModal.tsx (new frontend component)
- Revise Plan button in PlanReviewWorkspace
- Relax revise-thesis stage check to THESIS or PLAN

Out of scope:
- Revision history UI
- Revision count badge in other workspaces
- Revision from Approval stage or beyond
- Moving postReviseThesis() to runtime.ts

## Reference Pattern

Backend pattern: routes.py:1234–1373 (thesis history + revision)
Frontend pattern: ThesisRevisionModal.tsx
Stage validation pattern: routes.py ~line 1310

## Open Questions

None — pattern is unambiguous.
