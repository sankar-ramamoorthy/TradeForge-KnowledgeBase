# M10AIS04-05 Planning

## Scope

M10AIS04: ScenarioBranchArtifact domain model + create-scenario-branch endpoint + branches query endpoint
M10AIS05: ScenarioBranchPanel frontend + OpportunityWorkspace integration + ReplayWorkspace preview

## Core Concept

Scenario branches capture conditional reasoning — "if X then Y".
They prevent binary thinking and force explicit articulation of:
- primary scenario (what I expect to happen)
- alternative scenarios (what else might happen)
- invalidation pathways (what would prove me wrong)
- regime transitions (how a market shift changes the trade)

## Not A Lifecycle Transition

Scenario branches are enrichment events. Like thesis revisions, they:
- do NOT advance the lifecycle stage
- are appended directly to the event store (not via LifecycleOrchestrationService)
- can accumulate multiple branches on the same decision
- are valid at any active stage (Idea through Position, not after Review)

## ScenarioBranchArtifact Fields

- branch_type: str enum — "primary" | "alternative" | "invalidation" | "regime_transition"
- condition: str (required) — what triggers / when this branch applies
- implication: str (required) — what the operator does if this branch plays out
- confidence: int 1-5 (required) — how likely the operator expects this branch
- notes: str (optional) — additional context

## Event Model

New event type: `decision.scenario_branch_created`
Payload: {branch: {branch_type, condition, implication, confidence, notes}}
No new lifecycle stage — direct event store append

## Active Decision Check

Scenario branches valid when decision has at least one lifecycle event
AND current stage is not Review (completed decisions don't get new branches).

## Backend Endpoints

POST /lifecycle/decisions/create-scenario-branch
- validates branch fields (branch_type enum, condition, implication, confidence 1-5)
- checks decision exists and is not in Review stage
- appends decision.scenario_branch_created event directly

GET /lifecycle/decisions/{decision_id}/scenario-branches
- returns all scenario branches in chronological order
- groups: branch_type, condition, implication, confidence, notes, event_timestamp

## Frontend Changes

1. ScenarioBranchModal.tsx
   - branch_type select (Primary / Alternative / Invalidation / Regime Transition)
   - condition textarea (required)
   - implication textarea (required)
   - confidence slider (1-5)
   - notes textarea (optional)
   - submit to POST /lifecycle/decisions/create-scenario-branch
   - on success: re-fetch branches and close modal (stays in workspace)

2. ScenarioBranchPanel.tsx
   - Displays all branches for a decision
   - Grouped or labeled by branch_type with visual differentiation
   - Each branch: condition + implication + confidence badge + optional notes
   - "Add Scenario" button at the bottom
   - Absent when no branches exist AND no active decision

3. OpportunityWorkspace.tsx updates
   - Fetch scenario branches on load (GET /lifecycle/decisions/{id}/scenario-branches)
   - Show ScenarioBranchPanel below the field surfaces
   - "Add Scenario" button opens ScenarioBranchModal

4. ReplayWorkspace.tsx updates (M10AIS05)
   - Add extractScenarioBranchPayload() type guard
   - Add ScenarioBranchPreview component for timeline entries
   - Update CognitiveSnapshotSummary to include scenario branch count
   - TimelineEntryRow handles decision.scenario_branch_created events

## Testing Strategy

Backend:
- POST creates decision.scenario_branch_created event with structured payload
- POST rejects invalid branch_type
- POST rejects empty condition
- POST rejects confidence outside 1-5
- POST rejects after Review stage
- POST rejects non-existent decision
- GET returns all branches in chronological order
- Multiple branches accumulate correctly
- Different branch types are returned correctly
