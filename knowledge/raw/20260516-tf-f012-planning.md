# TF-F012 Planning - Workstation Layout Model

## Diagnosed Root Cause Summary

The frontend runtime translation layer still applies a centered document-style shell and a two-zone layout primitive. This no longer matches TradeForge's workspace doctrine or the newer operational layout drafts.

## Chosen Approach

Implement a staged frontend correction rather than a full-workspace redesign.

### Stage 1 - Shared Layout Primitive Alignment

- replace centered desktop shell assumptions for operational workspaces
- extend shared workspace layout primitives from:

```text
sidebar + main
```

to:

```text
navigation zone + primary operational zone + contextual awareness zone
```

- preserve current responsive collapse behavior where possible

### Stage 2 - Operating Workspace Reference Implementation

- convert the Operating Workspace first
- relocate persistent market/advisory/contextual material into the contextual awareness rail where appropriate
- restructure active operational objects so the primary surface can support more parallel scanning
- preserve existing lifecycle and authority language

### Stage 3 - Evaluate Follow-On Issues

After the reference implementation:

- inspect whether Opportunity, Plan Review, Active Position, Replay, and Review workspaces need separate TF-F issues
- do not absorb those changes into `TF-F012` unless the first implementation proves them mechanically required

## Why This Approach

This is the minimum change that tests the workstation model against real runtime surfaces without creating a frontend-wide redesign project.

It aligns with:

- bounded feedback-loop scope
- architectural simplicity
- operational verification against the original screenshot

## Affected Layers And Modules

Likely runtime files:

- `TradeForge/frontend/DESIGN.md`
- `TradeForge/frontend/src/styles.css`
- `TradeForge/frontend/src/operationalLayout.tsx`
- `TradeForge/frontend/src/workspaces/OperatingWorkspace.tsx`

## Lifecycle / Event / Replay Implications

- lifecycle implications: none
- event implications: none
- replay implications: none

This issue changes projection composition only. It must not alter canonical workflow authority.

## Testing Approach

Runtime verification should include:

- frontend typecheck
- frontend build
- screenshot/manual verification at desktop width against the original source observation
- responsive verification at narrow width to ensure safe degradation
- regression check that advisory/contextual surfaces remain visually distinguishable from canonical operational state

## Regression Risk Areas

- existing workspace shell alignment across all routes
- navigation overflow or collapse behavior
- context content becoming visually over-dominant
- mobile/narrow breakpoint regressions
- accidental advisory/canonical visual blending during layout refactor

## Out Of Scope

- resizable panels
- detached workspaces
- workspace presets
- multi-monitor support
- redesign of non-reference workspaces unless separately diagnosed

## Open Questions

- Should the contextual awareness rail become globally reusable immediately, or remain a shared primitive used only by the Operating Workspace first?
- Should `frontend/DESIGN.md` retain a shell width token at all, or redefine it only for non-operational/document-style surfaces?
- Which context panels belong in the initial right rail versus remaining inline during the first pass?
