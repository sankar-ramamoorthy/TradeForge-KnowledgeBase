# TF-F011 Planning — Docker Compose Fix

## Diagnosed Root Cause Summary

Placeholder `command` in `docker-compose.yml` causes the container to exit immediately. Server never runs inside Docker. `TRADEFORGE_DATABASE_URL` defined in compose never reaches the app. In-memory fallback activates silently. Events not persisted. Replayability broken.

## Chosen Approach and Rationale

Edit `docker-compose.yml` only. No application code changes needed.

The env var injection is already correct. The postgres service is correctly configured. The only missing piece is the server startup command.

Changes:

```yaml
# tradeforge service
command:
  [
    "uv", "run", "uvicorn",
    "src.app.api.application:app",
    "--host", "0.0.0.0",
    "--port", "8000"
  ]
ports:
  - "8000:8000"
restart: unless-stopped

# postgres service
restart: unless-stopped
```

The `src.app.api.application:app` module path is confirmed in the ISSUE_REGISTER for TF-0027 (FastAPI runtime), which verified the app starts with `uv run uvicorn src.app.api.application:app --host 127.0.0.1 --port 8000`.

## Scope Boundary

In scope:
- `docker-compose.yml` — command, ports, restart

Out of scope:
- `.env` file support
- Named bridge network
- Application-level event store logging

## Replay and Lifecycle Implications

No event model changes. No lifecycle changes. The fix restores the intended behavior — uvicorn starts with `TRADEFORGE_DATABASE_URL` in its environment, `create_event_store()` resolves to `PostgresEventStore`, events are written to `event_ledger`.

## Testing Approach

Verification after fix:
1. `docker compose up -d` — container stays running
2. `docker compose ps` — tradeforge service shows `Up`
3. `curl http://localhost:8000/health` — API responds
4. Create a decision via API
5. Query `event_ledger` directly — event persisted

Regression: `uv run pytest` to verify no code-level regressions (no code changed, but confirms environment integrity).

## Open Questions

None. Approach is clear and bounded.
