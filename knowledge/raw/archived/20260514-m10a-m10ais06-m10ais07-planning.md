# M10AIS06-07 Planning

## Scope

M10AIS06: TradePlanArtifact domain model + create-plan API endpoint + plan query endpoint
M10AIS07: PlanDevelopmentModal frontend form + PlanReviewWorkspace integration

Identical pattern to M10AIS01-02 (thesis). The Thesis→Plan transition is gated by
structured plan authoring.

## TradePlanArtifact Fields

- entry_rationale: str (required, min 10) — why this entry point and price level
- stop_rationale: str (required) — why this stop represents thesis invalidation
- target_rationale: str (required) — why this target represents thesis fulfillment
- sizing_rationale: str (required) — how position size was determined relative to conviction
- execution_assumptions: list[str] (required, min 1) — what must hold for the plan to execute
- playbook_alignment: str (optional) — which playbook this follows

## Event Model

Creates decision.plan_created event (existing lifecycle stage transition)
with structured plan payload: {plan: {entry_rationale, stop_rationale, target_rationale,
sizing_rationale, execution_assumptions, playbook_alignment}}

This IS a lifecycle transition (Thesis → Plan) via LifecycleOrchestrationService.

## ADR Checkpoint

No new ADR needed. ADR-0034 (Thesis And Plan Authoring Architecture) explicitly covers plan
authoring as the next step: "A similar POST /lifecycle/decisions/create-plan endpoint will gate
the Thesis→Plan transition."

## Backend Endpoints

POST /lifecycle/decisions/create-plan
- validates all required plan fields
- uses LifecycleOrchestrationService to create the lifecycle transition
- returns decision_id, event_type, timestamp

GET /lifecycle/decisions/{decision_id}/plan
- reads decision.plan_created event payload
- extracts TradePlanArtifact from payload
- returns 404 for decisions without a plan or with legacy empty-payload plan events

## Frontend Changes

1. PlanDevelopmentModal.tsx
   - entry_rationale (textarea, required)
   - stop_rationale (textarea, required)
   - target_rationale (textarea, required)
   - sizing_rationale (textarea, required)
   - execution_assumptions (dynamic list, required)
   - playbook_alignment (text input, optional)
   - submit to POST /lifecycle/decisions/create-plan
   - on success: reload projection (stay in plan-review, now at Plan stage)

2. PlanReviewWorkspace.tsx updates
   - "Create Plan" button opens PlanDevelopmentModal instead of empty transition
   - After plan creation: reload projection to show Plan stage actions
   - Add PlanContextPanel: shows plan content when lifecycle stage >= Plan
   - fetch plan artifact from GET /lifecycle/decisions/{id}/plan

## Key Difference from Thesis Flow

Thesis: on success navigate to plan-review
Plan: on success reload projection (stay in plan-review workspace — now at Plan stage)
This is because Plan stage leads to Authorize Plan (also in plan-review), not a new workspace.

## Testing Strategy

Backend:
- POST /lifecycle/decisions/create-plan creates decision.plan_created event with structured payload
- POST /lifecycle/decisions/create-plan rejects empty entry_rationale
- POST /lifecycle/decisions/create-plan rejects when not in Thesis stage
- POST /lifecycle/decisions/create-plan rejects empty execution_assumptions
- GET /lifecycle/decisions/{id}/plan returns plan content
- GET /lifecycle/decisions/{id}/plan returns 404 without a plan
- Plan payload embedded in replay timeline entries

Frontend:
- TypeScript typecheck clean
- Build clean
