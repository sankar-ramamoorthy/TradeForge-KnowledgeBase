# TF-F056 Planning - Missing Default Advisory Provider Bootstrap

## Diagnosed Root Cause Summary

`create_app()` references `_default_advisory_provider()` but the function is
missing. This causes Docker startup to fail with `NameError` before FastAPI can
serve any route.

## Chosen Approach

Restore the missing composition-root helper in `src/app/api/application.py`:

1. If no `CredentialStore` exists, return `None`.
2. If the `litellm` credential is not configured, return `None`.
3. If the master key is unavailable or the credential payload cannot be
   decrypted/validated, return `None`.
4. If credentials decrypt successfully, return
   `OpenAICompatibleAdvisoryProvider(litellm_credential)`.

This preserves the existing runtime behavior where advisory endpoints and health
checks report not configured/unavailable rather than startup failing.

## Scope Boundary

Only app composition and regression tests are in scope. No advisory service,
provider adapter, credential schema, Docker, or frontend behavior changes are
required.

## Event, Replay, And Lifecycle Implications

- Event impact: none.
- Replay impact: none.
- Lifecycle impact: none.
- AI advisory impact: provider construction remains non-authoritative and does
  not persist events or transition lifecycle state.

## Testing Approach

- Add focused tests for `create_app()` startup without LiteLLM credentials.
- Add focused tests for configured LiteLLM credentials resolving to
  `OpenAICompatibleAdvisoryProvider`.
- Run targeted app/advisory/credential tests.
- Run full backend test suite as the mandatory regression check.
- Run frontend typecheck/build because the feedback loop requires it, even
  though the fix is backend-only.

## Open Questions

- Whether future provider health should distinguish `not_configured` from
  `configured_but_key_unreadable`. That is out of scope for this startup crash.
