# TF-F070 Planning - Advisory Route Diagnostics And Explicit Advisory Invocation

## Chosen Approach

Implement TF-F070 as a bounded operational diagnostics enhancement using existing advisory boundaries.

## Runtime/API Changes

- Add a new provider governance AI gateway smoke-test endpoint that builds a tiny `AdvisoryRequest` with `artifact_kind=context-summary`, sends it through `AIAdvisoryService`, and returns a non-canonical operational diagnostic response.
- The smoke test must return configured/unavailable/not-configured style diagnostics without writing to the event ledger.
- Reuse the existing `AIAdvisoryProvider` and `AIAdvisoryService` validation path so advisory authority checks remain centralized.

## Frontend Changes

- Add `/advisory` to the Vite proxy table.
- Add an API helper for safe JSON parsing so HTML/proxy failures produce clear operational diagnostics instead of raw JSON parse errors.
- Add a thesis-review API helper and wire it to an explicit Plan Review action with idle/running/succeeded/failed state.
- Add a provider governance smoke-test button and display its operational result.

## Scope Boundary

- Generated advisory review content is displayed as a draft only. It is not captured into advisory interpretation storage in this issue.
- Smoke-test state is UI-local/operational only. It is not persisted and not replay-visible.
- Existing lifecycle transitions remain unchanged.

## Event And Replay Implications

- No new events.
- No event-ledger writes.
- Replay behavior unchanged.
- Advisory artifacts remain non-canonical and require separate operator acceptance before durable capture.

## Testing Approach

- Backend: add provider governance smoke-test tests proving success, not configured behavior, and no event-store mutation.
- Frontend: use TypeScript/build verification for runtime helper and UI wiring. Existing repo has no frontend unit-test framework.
- Config: verify Vite proxy includes `/advisory`.
