# DESIGN.md Introduction Timing

## Status

Processed synthesis from a raw note asking when TradeForge should introduce canonical `DESIGN.md`.

This note refines the timing and readiness criteria for `DESIGN.md`. It does not create canonical design doctrine.

The source raw note was processed and removed after this synthesis:

- `knowledge/raw/when would i introduce design.md`

## Core Synthesis

Canonical `DESIGN.md` should not be introduced before M5.

The current roadmap is still stabilizing foundational runtime semantics:

- event truth
- lifecycle authority
- replay determinism
- projection mechanics
- persona interpretation boundaries

A true TradeForge `DESIGN.md` would not describe screen appearance. It would describe behavioral architecture, experiential architecture, workflow cognition, and operational interaction philosophy.

Those concepts become coherent only after replay, workspaces, and persona projections exist.

```text
M5: Persona Workspace Projection Model
```

## Ontology Implications

This note reinforces that behavioral and experiential architecture is emergent from runtime-backed workspace and persona semantics.

After M5, the following concepts should be stable enough to support design doctrine:

- [[Workspace]] behavior
