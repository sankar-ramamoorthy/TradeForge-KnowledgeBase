The Four Remaining Decisions Are Actually Architectural Doctrine Decisions

These are not implementation details.

They shape the future operating philosophy of TradeForge.

1. Fundamentals rollout set

This is really deciding:

how much semantic heterogeneity enters the system initially

My recommendation:

One primary + one fallback

Why:

tests normalization discipline early
tests capability divergence
avoids overfitting to one provider
keeps complexity manageable
surfaces routing philosophy naturally

I would avoid:

all providers immediately
single-provider assumptions
2. Capability resolution model

This is extremely important.

My recommendation:

Preferred + ordered fallback

But:

deterministic
explicit
operator-visible
replay-recorded

Meaning:

Provider resolution is itself replayable operational context

That is critical.

Because provider choice changes:

visible information
advisory conclusions
workflow cognition
future replay interpretation

You absolutely do NOT want invisible dynamic routing later.

3. Operator surface

This is actually a sovereignty doctrine question.

My recommendation:

Editable configuration + visibility

Eventually:

health/status too

But not initially.

Operator should always know:

what provider was used
why
fallback sequence
missing capabilities
degraded capability state

Otherwise:

cognition provenance breaks
replay trust degrades
advisory traceability weakens
4. Fundamentals overlay placement

This is actually a workflow cognition question.

My recommendation:

Opportunity + Thesis initially

Not Plan yet.

Reason:

Fundamentals primarily influence:

idea qualification
thesis formation
contextual framing

By Plan stage:

operational intent should already be stabilizing

Putting rich contextual overlays too late risks:

thesis drift
plan instability
over-contextualization during execution prep

This aligns nicely with your lifecycle doctrine.
