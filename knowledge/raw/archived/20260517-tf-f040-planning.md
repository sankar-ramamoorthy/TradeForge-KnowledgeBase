# TF-F040 Planning - Trader-Language Boundary

## Goal

Define the boundary between canonical internal semantics and operator-facing UX language so later `M10E` work does not continue leaking runtime ontology into trader cognition surfaces.

## Architectural Interpretation

- Canonical ontology must remain stable in the domain model, glossary, events, and architecture docs.
- Operator-facing language should translate those semantics into terms that support trader cognition.
- The change belongs in doctrine and design architecture, with a runtime translation note for frontend implementation.
- No event, lifecycle, replay, or persistence behavior changes are involved.

## Planned Changes

1. Update canonical `UX_DOCTRINE.md` with an explicit trader-language principle.
2. Create canonical `design/trader-language-doctrine.md` with:
   - why the boundary exists
   - what should remain canonical internally
   - what should be translated externally
   - examples for headings, panels, empty states, and tooltips
3. Update `frontend/DESIGN.md` so runtime UI implementation checks copy against the trader-language rule.

## ADR Checkpoint

No ADR required.

Reason:

- this clarifies accepted UX doctrine rather than introducing a new bounded context
- no lifecycle, event, replay, or infrastructure boundary changes are introduced

## Validation

- canonical terms remain available internally
- operator-facing examples reduce ontology leakage
- later issues such as `TF-F029`, `TF-F035`, and `TF-F039` can reference a stable doctrine rule
