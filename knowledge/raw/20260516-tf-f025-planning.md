# Diagnosed Root Cause

The frontend treats runtime status as a panel concern rather than an application boundary. Downstream workspace components mount and request data before API availability is known.

# Chosen Approach

Use the existing `/health` request as the application gate:

- represent runtime loading, available, and unavailable states at `App` level
- render a focused unavailable surface while the API is down
- mount the operational layout only after health succeeds
- keep the existing runtime-boundary panel, but stop using it as the only health consumer

# Scope Boundary

Frontend-only change. No lifecycle, event, persistence, or backend contract changes.

# Replay And Lifecycle Implications

None. The fix changes only whether frontend projections are mounted before the API is reachable.

# Testing Approach

- factor runtime-gating presentation into testable behavior
- verify unavailable state does not render workspace surfaces
- verify available state renders the normal workspace shell

None at this stage.
