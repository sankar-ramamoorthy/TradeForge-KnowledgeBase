## Architectural Interpretation

TF-F004 is not a secret-storage implementation issue. It establishes the first stable runtime expression of the credential boundary already decided in ADR-0037.

The bounded runtime addition is:

1. Add `src/security/` as a top-level module boundary.
2. Add immutable `CredentialStatus` and `Credential` domain types.
3. Preserve provider-adapter independence from `src/security/`.
4. Avoid storage, decryption, composition-root wiring, and CLI work until TF-F005 through TF-F007.

## Event / Replay / Lifecycle Analysis

- Event impact: none in this issue. No new canonical events are introduced.
- Replay impact: the domain model carries `status`, `rotated_at`, and `last_validated_at` specifically so future replay-oriented credential status artifacts can be modeled without reshaping the entity.
- Lifecycle impact: none. Decision workflow semantics are unchanged.

## Design Plan

- `src/security/credential.py`
  - `CredentialStatus(StrEnum)` with `active`, `revoked`, `expired`, `unknown`
  - `Credential` as `@dataclass(frozen=True, slots=True)`
  - fields exactly matching ADR-0037 / issue acceptance criteria
  - lightweight validation for non-empty identifiers and timezone-consistent timestamp ordering where appropriate
  - normalize provenance to an immutable mapping to preserve the frozen domain boundary
- `src/security/__init__.py`
  - public exports for the new model
- `tests/test_credential.py`
  - enum value coverage
  - valid credential construction
  - validation failures for empty identifiers and invalid timestamp order
  - immutability check for provenance boundary
